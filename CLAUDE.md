# PoemGraph 项目规则

## 跨 AI 工具接手（强制）

本项目会在多个 AI 编程工具（不同 Claude Code 账号 / Codex 等）间切换开发。
- **接手第一步（省 token）**：会话开始**默认只读仓库根 `AGENTS.md` 的「当前状态 + 下一步 TODO」**，据此直接继续。
- **收尾**：日常只更新 `AGENTS.md`；PROJECT_PLAN / DESIGN 仅里程碑收尾才更新（见下「文档维护规则」）。
- 项目状态以仓库文件为准（可移植，跨账号生效），不依赖任何工具私有记忆（私有记忆不跨账号同步）。

### 接手省 token 协议（强制）
> 教训：曾因 AGENTS.md 膨胀 + dump 大 diff，光初始化就耗掉用户 ~26% token。

1. **分层只读**：接手只读 `AGENTS.md`（已精简，仅状态/TODO）。**实现细节按需读 `docs/IMPL_NOTES.md` 对应段**；变更历史用 `git log`（别读巨型 changelog）；PROJECT_PLAN/DESIGN 不预读。
2. **AGENTS.md 保持精简**：只放 状态/TODO/硬约束/文件地图，**变更日志不入 AGENTS**（进 git log/PROJECT_PLAN），实现备忘放 `docs/IMPL_NOTES.md`。每条 changelog/备忘**一句话**，禁长段落。
3. **工具调用省 token**：
   - 看改动用 `git diff --stat`，**绝不** `git diff 大文件` 或把整段日志/长行 grep 出来 dump；要看内容用窄 grep + `head -c` 限长。
   - 只 `Read` 需要的行段（offset/limit），不整文件读巨型代码；不全量扫码。
   - 验证优先 DOM 文字/`eval` 断言，截图节制（仅成品图）。

## 文档维护规则（强制执行 · 分层节奏省 token）

> 核心：**日常只动 `AGENTS.md`**；`PROJECT_PLAN.md` / `DESIGN.md` 只在**里程碑收尾**更新，避免每个小改动都写三份变更日志。

### 1. 日常（每个小改动 / 子功能完成）
- **只更新仓库根 `AGENTS.md`** 的「当前状态 / 下一步 TODO」（必要时「关键实现备忘」）。
- **不动** `PROJECT_PLAN.md` / `DESIGN.md`，不写它们的变更日志。

### 2. 里程碑收尾才更新（M 级里程碑 / 用户确认的大需求 / 架构决策）
- `docs/PROJECT_PLAN.md`：更新进度状态（`⬜ 待开始` → `🔄 进行中` → `✅ 已完成`）+ 在`## 变更日志`追加一行。
- `docs/DESIGN.md`：更新对应章节使其反映**当前最新**产品定义（不保留历史草稿）+ 在`## 变更日志`追加一行。
- 触发判断：是否新增/改变了一个完整功能模块、用户确认的新需求、或架构决策 → 是则属里程碑，更新这两份；否则只算日常。

### 3. 每次对话结束前检查清单
- [ ] `AGENTS.md` 是否最新（当前状态 / 下一步 TODO）？
- [ ] **本次是否达里程碑？** 若是 → PROJECT_PLAN / DESIGN 是否已更新并各追加一行变更日志？若否 → 跳过这两份。

## 里程碑测试规则（强制执行）

- **每完成一个里程碑（M0–M10）的修改后，必须立即做一次全面测试**，不得跳过。
- 测试内容至少覆盖：断网离线可用、五入口与详情页可正常渲染、本里程碑新增/改动功能、控制台无报错、水墨基调（配色可演进、含朱砂红）、iPad/桌面响应式。
- 测试使用 Claude Preview / 浏览器 MCP，不得让用户手动验证。
- **截图节制省 token**：验证以 **DOM 文字快照 / `eval` 取值**为主（断言元素、文本、状态、控制台、网络，都走文字工具），**不逐步截图**；截图仅在最终向用户展示成果时给**一张「成品证明图」**。
- 测试通过后再在 PROJECT_PLAN.md 标记该里程碑为 ✅，并在变更日志记录"已测试"。

## 视觉风格（强制约束）

**风格：水墨画（Ink Wash）**，以下规则不可违反：
- 背景色：宣纸白 `#f7f5f0`，不使用纯白或彩色背景
- 主文字：浓墨 `#1c1c1e`，副文字 `#5c5c6e`
- **配色无禁红限制（2026-06-22 取消禁红规则）**：朱砂红 / 朱印红（vermilion，如 `#c1352b`/`#b23a2e`）可自由用于强调、印章、朱批、点缀乃至更大占比（传统诗集本就用朱印朱批，古典且吸引小学生）。**视觉正用 `/frontend-design` 重设计中**（详情页样例待确认后推广）。
- 强调色：墨青 `#2d5a6b` + 朱砂红 暖色,可适度提高占比(不再硬性 <15%)
- 图谱节点用墨色系（见 DESIGN.md 第五节）
- 卡片背景：`#efece4`，边框：`1px solid #d8d4c8`
- 阴影：`box-shadow: 0 2px 12px rgba(28,28,30,.08)`，不用彩色阴影
- 装饰元素：水墨山形、竹叶 SVG，不使用彩色插图

## 表达原则（强制）：图形优先于文字

> 用户群体是小学生及一般用户，对图像/动效的接受度远高于文字阅读。**能用图形表达的，就不要用文字。**

- **能图形化就图形化**：方向→流动虚线（蚂蚁线）/箭头；顺序→编号徽章/起止配色（起点绿、终点赭）；数量→大小/数目；状态→颜色/图标；关系→连线。**不要**用整句文字注解去解释画面里已能看出的东西（如曾经的"足迹（①→②顺序）"——已删除，改用流动虚线表方向）。
- **复杂逻辑用「图形 + 文字」**：当纯图形会产生歧义或信息不全时，才辅以**最短**文字标签（如地名、诗题、编号），而非整句说明。
- **动效优先于说明**：能用一个动效让用户"看懂"的，就不要写一句话让用户"读懂"。
- **删冗余文字**：画面中与图形重复、或仅作说明用途的长文字，一律精简或删除。
- 适用范围：地图、知识图谱、时间轴、详情页、练习等所有可视模块。

## 技术规范

- **交付物**：单一 HTML 文件 `poemgraph.html`，无外部依赖，完全离线可用
- **数据存储**：所有诗词数据内嵌在 HTML 的 JS 对象中；用户进度/偏好用 `localStorage`
- **图谱库**：不使用外部库，用原生 Canvas 自绘力导向图（离线零依赖）
- **3D模式**：Canvas 伪 3D（球形节点透视旋转），不内联 Three.js，控制文件体积
- **响应式**：支持 iPad（768px+）和桌面（1024px+）

## 内容规范

- 诗词数据与**统编版**小学语文课本对齐（1–6年级）
- **内容优先来源**：统编版教材 PDF（GitHub: wen001-git/ChinaTextbook，路径 `小学/语文/统编版`），原文、拼音、字词注释以教材为准
- 诗词数据与人教版小学语文课本对齐（1–6年级）
- 每首诗必须包含：原文+拼音、诗意讲解、诗人故事、历史背景、地理位置、趣味测验（4题）
- 诗意讲解用小学生能懂的语言，按年级调整深度
